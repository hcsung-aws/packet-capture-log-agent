# 리플레이 시스템 설계 패턴

## 코어 루프 / 데이터 처리 분리

리플레이 기능 구현 시 아래 두 관심사를 반드시 분리:

| 관심사 | 책임 | 변경 빈도 |
|--------|------|----------|
| 코어 루프 | 연결, 타이밍, 송수신 시퀀싱 | 낮음 |
| 데이터 처리 | 응답 해석, 필드 주입, 로깅 | 높음 |

분리하지 않으면 새 변형(파싱 추가, 필드 주입 등)마다 코어 루프를 복제하게 됨.

구현 방법: 인터페이스/콜백으로 데이터 처리를 주입.
```
Replay(packets, handler)  // handler가 응답 처리 담당
```

## 노이즈 패킷 이중 분류

게임 패킷 분석 시 노이즈를 단순 "제외"로 처리하면 안 됨:

| 분류 | 설명 | 예시 |
|------|------|------|
| replay_skip | 리플레이 시 전송 불필요 | SC_NPC_SPAWN, SC_CHAR_LEAVE |
| data_source | 리플레이 불필요하지만 데이터 추출 필요 | SC_NPC_SPAWN (npcUid → 공격 대상) |
| pure_noise | 완전 무시 가능 | SC_HEARTBEAT |

data_source 패킷은 수신 시 상태에 반영만 하고 리플레이 시퀀스에는 포함하지 않음.
